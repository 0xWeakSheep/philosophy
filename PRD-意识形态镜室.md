## 1. Summary

意识形态镜室是一个 AI 自我认知产品。它用 4×4 哲学分析框架，引导用户看见自己相信什么、在乎什么、卡在哪里、能往哪走。

v1 先聚焦一个最小可验证场景：关系困境。用户从一个轻量入口开始，进入框架驱动的追问，最后得到一张可回溯的自我意识形态地图。

产品不做心理诊断，不做治疗建议，不做人格标签，也不把用户归类为某种政治或哲学立场。

## 3. Background

### 3.1 用户问题

很多自我认知工具只能停在表层。情绪日记可以记录当下感受，但很难追问用户为什么总是用同一套方式理解关系。人格测评能给标签，但标签容易让人停止思考。通用 AI 聊天能陪用户说很多话，但缺少稳定框架，容易变成共情、建议或绕圈。

意识形态镜室要解决的问题是：用户反复遇到相似困境，却看不清自己背后的意义结构。比如，一个人总在亲密关系里感到被否定。表层问题是沟通冲突。更底层的问题可能是：他默认相信关系应该遵循什么秩序，什么东西对他来说不可妥协，他如何解释自己的感受，他的行动到底在保护什么。

### 3.2 为什么现在做

现在具备三个条件：

1. LLM 已经能进行多轮对话，可以承载 30 到 60 分钟的深度探索。
2. 用户已经熟悉 AI 聊天，但需要更有方向的深度产品。
3. 本项目已有一套 4×4 知识库，可以把抽象哲学框架转成产品追问、坐标解释和跃迁路径。

### 3.3 核心框架

产品使用四个维度理解用户的思想姿态：

| 维度 | 核心问题 | 产品转译 |
|---|---|---|
| 场域论 | 世界或关系场是什么样的？ | 用户默认相信哪些规则、边界和“当然如此”。 |
| 本体论 | 什么是真的、不可绕开的？ | 用户把什么视为硬核事实、底层机制或不可妥协之物。 |
| 现象论 | 这一切如何被体验？ | 用户怎样感受、命名、解释和组织自己的经验。 |
| 目的论 | 最终朝哪里运动？ | 用户的行动、欲望、逃避和选择指向什么。 |

每个维度有四种要素：秩序、冲突、中心、虚无开放。坐标不是人格类型，也不是高低分。它只是一个临时假设，用来帮助用户看清当前议题里的思想结构。

### 3.4 产品边界

v1 必须守住这些边界：

1. 不诊断用户。
2. 不提供临床治疗建议。
3. 不说“你就是某某主义者”。
4. 不把 `4` 说成天然优于 `1`。
5. 不做哲学课程。
6. 不做社交和社区。
7. 不把知识库内容直接堆给用户。

更合适的表达方式是：“在这个议题里，你的回答暂时呈现出某种坐标倾向。”

## 4. Objective

### 4.1 产品目标

v1 的目标是跑通一次完整的自我探索旅程：

1. 用户在 5 分钟内完成一个低门槛入口。
2. 用户进入 30 到 60 分钟的框架驱动对话。
3. AI 通过追问帮助用户发现自己的临时坐标。
4. 用户获得一张可保存、可回看的自我意识形态地图。
5. 用户知道这个坐标的盲区，以及下一次可以探索什么。

### 4.2 公司目标

验证“哲学框架驱动的 AI 自我认知”是否能成立为一个独特产品，而不是普通 AI 聊天、人格测评或心理工具的变体。

### 4.3 用户收益

用户不只是得到建议，而是能看见自己如何制造、维持或解释困境。用户离开产品时，应带走一个可以继续思考的问题结构，而不是一个简单标签。

### 4.4 Key Results

| 指标 | 当前状态 | v1 目标 | 测量方式 |
|---|---|---|---|
| 入口完成率 | 未建立基线 | 首批测试用户中，70% 完成 5 分钟入口 | 事件埋点：入口开始到入口完成 |
| 完整旅程完成率 | 未建立基线 | 首批测试用户中，50% 完成一次 30 到 60 分钟探索 | 事件埋点：入口完成到显影完成 |
| 认知发现感 | 未建立基线 | 60% 完成用户选择“我看见了以前没注意到的问题结构” | 显影后单题反馈 |
| 非标签化感受 | 未建立基线 | 80% 完成用户不同意“产品在给我贴标签” | 显影后单题反馈 |
| 安全边界合格率 | 未建立基线 | 抽样会话中，95% 不出现诊断、治疗承诺、政治定性或人格定型 | 人工评审与自动评测 |
| 回访意愿 | 未建立基线 | 40% 完成用户在 14 天内回看地图或开启新议题 | 留存与行为事件 |

### 4.5 Non-Goals

v1 不做以下内容：

1. 不做社交、社区、匹配或公开分享。
2. 不做心理测评报告或临床建议。
3. 不做课程、文章流、哲学术语学习。
4. 不覆盖所有人生议题。v1 先做关系困境。
5. 不做“最终人格档案”。每次坐标都只绑定具体议题和时间。

## 5. Market Segment(s)

### 5.1 首要用户

首要用户是有自我探索意愿的人。他们常见特征是：

1. 反复遇到相似的关系困境。
2. 不满足于情绪记录、性格测试或泛泛建议。
3. 愿意被追问，但不希望被说教。
4. 能接受一次 30 到 60 分钟的深度对话。
5. 没有把产品当作临床治疗入口。

典型任务：

1. “我为什么总在关系里重复同一种模式？”
2. “我知道自己难受，但说不清真正卡在哪里。”
3. “我不想要建议，我想知道我为什么会这样理解这件事。”

### 5.2 次要用户

次要用户包括：

1. 对哲学、心理学、自我成长内容有兴趣，但不想上课的人。
2. 需要梳理重大选择的人，比如亲密关系、职业转向、家庭边界。
3. 高反思密度的知识工作者、创作者、咨询从业者。

### 5.3 不适合的用户

v1 不面向这些场景：

1. 正处在心理危机、创伤急性期或自伤风险中的用户。
2. 期待产品给出诊断、处方、治疗计划的人。
3. 只想快速得到行动建议，不愿做反思的人。
4. 想学习未明子体系、拉康、康德或哲学史的人。

## 6. Value Proposition(s)

### 6.1 用户要完成的工作

用户不是来获得一个答案，而是来完成三件事：

1. 把混乱经历说出来。
2. 看清自己默认的解释方式。
3. 找到一个可以继续行动的新问题。

### 6.2 用户获得什么

| 用户收益 | 具体表现 |
|---|---|
| 看见默认规则 | 用户能说出自己在关系里默认相信的“应该如此”。 |
| 看见不可妥协物 | 用户能发现自己真正保护的东西。 |
| 看见体验结构 | 用户能区分事实、解释、情绪和旧叙事。 |
| 看见行动方向 | 用户能理解自己是在维持秩序、制造冲突、寻找中心，还是停在开放里。 |
| 看见变化路径 | 用户能知道下一次探索可以从哪里开始。 |

### 6.3 用户避免什么痛苦

1. 避免被简单标签固定。
2. 避免只收到“你应该怎么做”的建议。
3. 避免在同一个故事里反复讲述，却没有结构变化。
4. 避免被哲学术语吓退。
5. 避免把自我探索误认为心理诊断。

### 6.4 相比替代方案的差异

| 替代方案 | 常见问题 | 意识形态镜室的差异 |
|---|---|---|
| 情绪日记 | 记录感受，但不一定追问结构 | 把感受放进场域、本体、现象、目的四个维度中看。 |
| 人格测评 | 给静态标签，用户容易自我固化 | 给具体议题下的临时坐标，并允许随时间变化。 |
| 通用 AI 聊天 | 对话自由，但方向不稳定 | 背后有知识库和追问策略，避免泛泛共情。 |
| 心理咨询 | 深度高，但成本高，也不是所有用户都需要 | 提供非临床的自我探索入口，并明确风险边界。 |
| 哲学内容 | 信息多，但用户要自己转译 | 框架在幕后工作，用户只感受到好问题。 |

## 7. Solution

### 7.1 UX / Prototype

v1 的完整用户旅程如下：

```text
边界说明
  -> 选择议题：关系困境
  -> 5 分钟入口：场景选择 / 卡牌排序 / 自由书写
  -> AI 形成临时坐标假设
  -> 30 到 60 分钟追问
  -> 用户确认或修正关键发现
  -> 显影：4×4 自我地图
  -> 保存版本
  -> 给出盲区和下一次探索入口
```

#### 7.1.1 边界说明

用户开始前必须看到简短说明：

1. 这是自我探索产品，不是心理治疗。
2. AI 不会诊断你。
3. 坐标是临时假设，不是人格标签。
4. 如果你有急性危机，应联系专业支持。

用户确认后才能进入。

#### 7.1.2 入口

入口必须轻、短、直觉化。v1 支持一种主入口，预留另外两种。

P0 入口：自由书写。

提示语示例：

> 写下一个你反复遇到的关系困境。不用解释得很完整。只要写出发生了什么、你最难受的地方、你通常会怎么做。

P1 入口：情境选择。

用户在几个关系场景中选择最像自己的一个。

P1 入口：卡牌排序。

用户把“秩序、冲突、中心、开放”等经过产品化改写的卡牌按直觉排序。

#### 7.1.3 深度对话

AI 不能直接给结论。它必须先追问，再形成假设。追问顺序不固定，但必须覆盖四个维度：

1. 场域论：你默认认为关系应该遵循什么规则？
2. 本体论：这件事里，你最不能让步的东西是什么？
3. 现象论：你如何感受和解释这件事？
4. 目的论：你反复这样做，实际在保护什么或走向什么？

AI 每隔若干轮可以做一次简短回看：

> 我先不下结论。到目前为止，我听到三个线索：你很在意关系里的稳定规则，你把某种承诺视为不可退让的真实，同时你会把自己的不安翻译成“我是不是做错了”。这三个线索可能指向一个临时坐标。我们再看一个问题。

#### 7.1.4 显影地图

显影页必须包含：

1. 当前议题。
2. 时间。
3. 临时坐标，例如 `1-2-3-1`。
4. 四个维度的解释。
5. 每个维度的证据句，来自用户原话或概括。
6. 盲区。
7. 可能跃迁方向。
8. 下一次探索问题。

显影页禁止出现：

1. “你就是某某主义者”。
2. “你应该成为更高级的坐标”。
3. 临床诊断。
4. 政治立场判定。
5. 绝对化建议。

### 7.2 Key Features and Requirements

#### P0 Must Have

| 编号 | 用户故事 | 验收标准 |
|---|---|---|
| P0-1 | 作为用户，我想知道产品边界，这样我不会把它误认为治疗或测评。 | 开始前显示边界说明；用户必须确认；文案不超过 120 字；包含非诊断、非治疗、临时坐标三点。 |
| P0-2 | 作为用户，我想用简单方式输入一个关系困境，这样我能快速开始。 | 用户可在 5 分钟内完成自由书写；系统能提取议题、关键关系、痛点、常见行动。 |
| P0-3 | 作为用户，我想被 AI 有方向地追问，而不是被直接评价。 | AI 至少覆盖四个维度；每个维度至少有一个开放问题；不得在前 5 轮直接给坐标结论。 |
| P0-4 | 作为用户，我想看到自己的临时坐标，这样我能理解当前困境的结构。 | 显影页展示四位坐标；每一位都有通俗解释；每一位都有证据句。 |
| P0-5 | 作为用户，我想知道这个坐标的盲区，而不是只得到描述。 | 显影页至少给出 2 个盲区；盲区必须写成问题，不写成批判。 |
| P0-6 | 作为用户，我想保存这次探索，这样以后可以回看变化。 | 每次探索生成一个版本；版本包含议题、日期、坐标、摘要、盲区、下一问。 |
| P0-7 | 作为产品团队，我想限制高风险输出，这样产品不越过心理健康和政治分类边界。 | 自动检测诊断、治疗承诺、自伤危机、政治定性、人格定型；命中时转为安全回应或危机提示。 |
| P0-8 | 作为产品团队，我想评估追问质量，这样可以持续改进 prompt。 | 每次会话保存匿名化质量指标；人工抽样可看到问题覆盖、坐标证据、安全违规。 |

#### P1 Should Have

| 编号 | 用户故事 | 验收标准 |
|---|---|---|
| P1-1 | 作为用户，我想选择不同入口方式，这样开始体验更自然。 | 支持自由书写、情境选择、卡牌排序三种入口；三种入口都能进入同一坐标推断流程。 |
| P1-2 | 作为用户，我想比较两次探索，这样我能看见变化。 | 版本页能并排展示两次坐标、议题、盲区和下一问。 |
| P1-3 | 作为用户，我想导出一份简洁报告，这样我可以离线保存。 | 支持 Markdown 或 PDF 导出；导出内容包含边界声明。 |
| P1-4 | 作为产品团队，我想测试不同追问策略，这样能找到更有效的路径。 | 支持 prompt 版本号；会话记录关联 prompt 版本；能比较完成率和用户反馈。 |

#### P2 Future

| 编号 | 用户故事 | 验收标准 |
|---|---|---|
| P2-1 | 作为用户，我想探索职业、家庭、自我价值等议题。 | 新议题有独立入口题库和评估集；不影响关系困境主流程。 |
| P2-2 | 作为用户，我想听到更自然的语音追问。 | 语音模式仍遵守同样安全边界；可随时切回文字。 |
| P2-3 | 作为用户，我想得到轻量行动实验。 | 行动实验必须是反思性任务，不是治疗计划；每个实验绑定一个盲区。 |
| P2-4 | 作为团队，我想沉淀匿名模式洞察。 | 只使用用户授权的匿名数据；不得反推出个人身份。 |

### 7.3 Technology

#### 7.3.1 系统组件

| 组件 | 职责 |
|---|---|
| 前端应用 | 提供入口、对话、显影地图、版本历史。 |
| Conversation Orchestrator | 管理多轮对话状态、追问节奏、阶段切换。 |
| Knowledge Retrieval | 从 4×4 知识库读取维度、要素、坐标、破解路径。 |
| Coordinate Hypothesis Engine | 根据用户表达形成临时坐标假设，并记录证据与置信度。 |
| Safety Layer | 识别诊断、治疗建议、自伤危机、政治定性和人格定型风险。 |
| Session Store | 保存用户会话、版本记录、反馈和匿名质量指标。 |
| Evaluation Harness | 用固定样本测试 prompt、坐标解释和安全输出。 |

#### 7.3.2 坐标推断原则

坐标推断不能只靠一次分类。它应满足四个要求：

1. 每一位坐标都必须有证据。
2. 证据必须来自用户表达，不来自 AI 猜测。
3. 置信度低时，AI 必须继续追问。
4. 输出必须保留“临时”和“可修正”的语气。

推荐内部数据结构：

| 字段 | 说明 |
|---|---|
| `session_id` | 一次探索的 ID。 |
| `topic` | 用户本次探索的议题。 |
| `dimension` | 场域论、本体论、现象论、目的论之一。 |
| `candidate_value` | 秩序、冲突、中心、虚无开放之一。 |
| `evidence` | 支撑该判断的用户表达。 |
| `confidence` | 低、中、高。 |
| `question_next` | 下一步追问。 |
| `risk_flags` | 安全风险标记。 |

#### 7.3.3 安全与隐私

P0 必须支持：

1. 明确告知产品不是心理治疗。
2. 对自伤或伤人风险给出危机支持提示。
3. 不生成诊断名、治疗方案或用药建议。
4. 不把用户归类为政治群体或哲学阵营。
5. 用户可以删除会话和版本记录。
6. 内部评测使用匿名化数据。

### 7.4 Assumptions

| 假设 | 风险 | 验证方式 |
|---|---|---|
| 用户愿意花 30 到 60 分钟完成一次深度探索。 | 旅程太长，完成率低。 | 小规模可用性测试，观察中途退出点。 |
| 关系困境是最适合 v1 的入口。 | 用户问题太多样，单一场景不够吸引。 | 招募不同用户，比较关系、职业、家庭三个入口的启动率。 |
| 4×4 框架可以被产品化成用户听得懂的问题。 | 哲学味太重，用户被吓退。 | 对入口文案和显影解释做理解度测试。 |
| LLM 可以稳定执行苏格拉底式追问。 | 对话滑向共情、建议或说教。 | 建立评测集，人工评审每轮追问是否有效。 |
| 坐标输出不会让用户觉得被贴标签。 | 用户把坐标当成人格判定。 | 显影后询问“是否感觉被贴标签”，并测试不同文案。 |
| 知识库足够支撑 v1。 | 某些坐标解释太抽象，不能转成追问。 | 针对关系困境补齐高频坐标的追问模板和边界句。 |

### 7.5 Open Questions

| 问题 | Owner | 建议解决时间 |
|---|---|---|
| v1 是否只做关系困境，还是允许用户自由输入任意人生议题？ | Product Owner | Demo 前 |
| 坐标显影是一次性展示，还是边对话边逐步亮起？ | Design Owner | Demo 前 |
| 置信度低时，是继续追问，还是允许输出多个候选坐标？ | AI/Prompt Owner | MVP 前 |
| 用户是否需要账号才能保存版本？ | Engineering Owner | MVP 前 |
| 危机提示文案和升级路径如何设计？ | Legal/Safety Reviewer | MVP 前 |
| 哪些会话数据可以用于匿名评测？ | Legal/Safety Reviewer | MVP 前 |

## 8. Release

### 8.1 V0 Demo

目标：跑通一条可演示的完整路径。

范围：

1. 单一入口：关系困境自由书写。
2. 单一对话流程：四维追问。
3. 简单显影页：坐标、解释、证据、盲区、下一问。
4. 手动或半自动版本保存。
5. 基础安全提示。

不做：

1. 多入口。
2. 多议题。
3. 账号系统。
4. 导出报告。
5. 复杂可视化。

### 8.2 MVP

目标：让首批测试用户能独立完成一次探索。

范围：

1. 完整边界说明。
2. 关系困境入口。
3. 框架驱动对话。
4. 坐标推断与证据记录。
5. 显影地图。
6. 版本历史。
7. 用户反馈。
8. 安全层和人工抽样评审。

MVP 完成标准：

1. 用户不需要产品团队陪同，也能完成一次探索。
2. 显影页能被用户看懂。
3. 坐标解释不会被多数用户理解为人格标签。
4. 安全评审没有明显越界输出。
5. 团队能从会话中看到下一轮 prompt 和知识库改进点。

### 8.3 Private Beta

目标：验证产品是否值得继续做。

范围：

1. 招募一批有自我探索动机的用户。
2. 观察入口完成率、完整旅程完成率、回访意愿。
3. 对所有低分会话做人工复盘。
4. 根据反馈调整入口、追问和显影文案。

Beta 后决策：

1. 如果用户完成率低，先缩短流程或调整入口。
2. 如果用户觉得被贴标签，重写显影页和坐标语气。
3. 如果用户只获得“有趣”但没有“发现”，加强追问策略和盲区生成。
4. 如果用户有强回访意愿，再扩展第二个议题场景。

### 8.4 Future Versions

后续版本可以考虑：

1. 职业选择、家庭关系、自我价值等新议题。
2. 多次探索的轨迹图。
3. 轻量行动实验。
4. Markdown 或 PDF 报告导出。
5. 语音对话。
6. 经过用户授权的匿名模式分析。

这些都不应改变 v1 的核心边界：产品帮助用户看见问题结构，不替用户下诊断，不替用户做人生决定。
